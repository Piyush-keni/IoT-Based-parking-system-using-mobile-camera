Abstract

SmartPark Pro is an IoT system automating urban parking through number plate recognition and real-time slot monitoring. The hardware backbone uses an Arduino Uno R3 paired with LDR sensors for occupancy detection and a Servo SG90 for gate control. After a customer pre-books via the web portal, a Python + EasyOCR vision pipeline reads the arriving vehicle's plate, looks it up in Google Sheets, and signals the Arduino to open the gate and activate the guide LED. The project is validated on a 3-slot test bench and maps to SDG 9 (Innovation) and SDG 11 (Sustainable Cities).

Arduino Uno R3 LDR Sensor Servo SG90 Python EasyOCR DroidCam Google Sheets API
Problem Statement

Urban parking suffers from inefficiency and congestion. Drivers waste time searching for free slots while operators lack real-time visibility into occupancy — leading to traffic build-up and revenue loss.

Project Objectives
  • Automate slot allocation via OCR number plate recognition
  • Real-time LDR-based occupancy detection (100 ms polling)
  • Cloud database integration for live slot updates
  • QR-code pre-booking via web portal

Algorithm / Implementation Steps
  1. 01
    Customer Pre-Books Online
    User visits the web portal, selects a free slot (live from Google Sheets), enters name + vehicle number, pays ₹50. Booking is written to the cloud database and a QR code is generated from the plate number.
  2. 02
    Vehicle Arrives — OCR Scans Plate
    DroidCam app streams a live video feed from an Android phone to the Python script running on the PC. EasyOCR extracts the number plate text and a regex cleans it to standard format (e.g. MH12AB1234).
  3. 03
    Plate Verified Against Database
    The extracted plate string is looked up in Google Sheets via the Apps Script API. If a matching pre-booking exists → verified ✅. If not → access denied, user alerted ❌.
  4. 04
    Gate Opens + LED Guide Activates
    Arduino receives a serial command. Servo SG90 rotates to open the barrier (~3 s after verification). The corresponding slot LED turns on to guide the driver to the correct spot.
  5. 05
    LDR Monitors Slot Occupancy
    Each LDR sits in a voltage-divider circuit on analog pins A0–A2. Arduino polls every 100 ms. When the LDR is covered (car present) → slot is marked OCCUPIED in the DB and shown on the web portal.
  6. 06
    Vehicle Exits — Database Updated
    LDR goes uncovered → Arduino signals exit. Gate closes (servo returns, auto-closes 5 s after opening), guide LED turns off, Google Sheets row is cleared, and slot appears FREE on the portal within 30 s.

Overview

The SmartPark Pro hardware was designed end-to-end in KiCad EDA — from schematic capture through PCB layout. The board integrates the Arduino Uno R3 footprint, three LDR sensor circuits, three LED indicators with current-limiting resistors, a servo connector, and a decoupling capacitor — all routed on a single copper layer for straightforward fabrication.

KiCad EDA Single-Layer PCB Through-Hole (THT) Arduino Footprint Gerber Export
Schematic Design
Fig 1 — KiCad Schematic Editor: Arduino Uno R3 · 3× LDR voltage dividers (R4/R5/R6 = 10kΩ) on A0–A2 · 3× LEDs (D1/D2/D3) with 330Ω resistors (R7/R8/R9) · Servo header J1 · Decoupling cap C1 · PWR_FLAG net tie
Schematic Component Breakdown
  • 3× LDR sensors in voltage-divider configuration — LDR in series with 10kΩ pull-down (R4, R5, R6), mid-point to analog inputs A0, A1, A2
  • 3× LEDs (D1, D2, D3) with 330Ω current-limiting resistors (R7, R8, R9) connected to digital output pins
  • Servo SG90 3-pin header (J1) on PWM-capable digital pin for gate barrier control
  • Polarised electrolytic capacitor C1 across VCC/GND for power supply decoupling and noise filtering
  • PWR_FLAG symbols on both VCC and GND nets for clean ERC compliance in KiCad
PCB Layout (KiCad PCB Editor)
Fig 2 — KiCad PCB Layout (F.Cu layer, blue traces): Arduino Uno R3 footprint centre-right · 3× LDR modules arranged in left column · 3× LED + resistor pairs at bottom-right · Servo header and cap at top
Layout Design Decisions
  • Arduino Uno R3 (A1) placed centrally to keep trace lengths short to all peripheral components
  • LDR sensor pads (LDR1, LDR2, LDR3) arranged in a vertical left column mirroring the 3 physical parking slots
  • LED indicators (D1, D2, D3) and resistors grouped at bottom-right for easy panel-mount access
  • Servo connector (J1) and decoupling capacitor (C1) placed at top for clean power entry routing
  • All traces routed on F.Cu (front copper) single layer — no vias required, suitable for DIY etching or low-cost fab
AI-Generated Ideas for PCB Enhancement

As an AI assistant, here are some innovative ideas to improve the SmartPark Pro PCB design for better performance, scalability, and manufacturability:

  • Multi-Layer Transition: Upgrade to a 2-layer PCB for better EMI shielding and ground plane integrity, reducing noise in LDR readings
  • Integrated Power Regulation: Add an LM7805 voltage regulator footprint for stable 5V supply, eliminating reliance on Arduino's onboard regulator
  • Expansion Headers: Include I2C and SPI headers for future sensor additions like ultrasonic distance sensors or RFID readers
  • LED Current Limiting: Replace fixed resistors with PWM-controlled MOSFETs for adjustable LED brightness based on ambient light
  • Debugging Features: Add test points and a programming header for easier firmware updates without removing the Arduino
  • Thermal Management: Include heatsinks or thermal vias near power components to handle extended operation

PCB Design Workflow
  1. 01
    Schematic Capture in KiCad Schematic Editor
    Placed Arduino Uno R3 symbol, LDR + resistor voltage-divider subcircuits, LED + resistor output stages, servo header, power symbols (VCC, GND, PWR_FLAG), and net labels for all connections.
  2. 02
    ERC Check + Netlist Generation
    Ran Electrical Rules Check (ERC) to catch unconnected pins and floating power nets. Added PWR_FLAG net ties to resolve power warnings. Exported the netlist for import into PCB Editor.
  3. 03
    Footprint Assignment
    Assigned KiCad footprint library entries to every symbol: Arduino_UNO_R3 for the MCU, Conn_01x03_Pin for the servo header, through-hole pads for LDRs, LEDs, and axial resistors.
  4. 04
    Component Placement
    Arranged footprints logically on the board with the Arduino centred, LDR modules in a column on the left to reflect the physical slot layout, and LEDs grouped for easy indicator visibility.
  5. 05
    Manual Trace Routing on F.Cu
    Routed all signal and power connections on the front copper layer. VCC and GND traces given wider width for current capacity. Maintained ≥0.25 mm clearance throughout.
  6. 06
    DRC + Gerber File Export
    Design Rule Check (DRC) passed with zero clearance violations. Exported Gerber files (F.Cu, Edge.Cuts, F.Silkscreen) and drill file — ready for PCB fabrication or in-house etching.

Trade 03 / Website Designing

Overview

The SmartPark Pro web portal is the customer-facing booking interface built with vanilla HTML5, CSS3, and ES6+ JavaScript — no frameworks, no build tools. It connects to a Google Apps Script backend that reads and writes to Google Sheets as a free cloud database. On payment confirmation, a QR code is generated from the vehicle plate number, which the driver shows at the gate for OCR verification.

HTML5 CSS3 Vanilla JS ES6+ Google Apps Script Google Sheets QRCode.js Fetch API Python Flask
Key Features
  • Live slot availability from Google Sheets, auto-refreshed every 30 s
  • Interactive slot cards showing FREE / OCCUPIED with real-time DOM updates
  • Form validation — name required, vehicle plate regex enforced (MH12AB1234 format)
  • QR code generated from plate number shown instantly after payment
  • Silent background backup POST to local Flask server (bookings.txt)
  • Responsive layout — single-column slots on mobile (<500px breakpoint)
Tech Stack
  • Structure: HTML5 with semantic markup
  • Styles: CSS3 — custom properties, Grid, Flexbox, transitions
  • Logic: Vanilla JS — async/await, Fetch API, DOM manipulation
  • Backend: Google Apps Script deployed as Web App (GET/POST)
  • Database: Google Sheets — free, serverless, cloud-synced
  • Libraries: qrcode.js v1.4.4 via CDN (canvas-based QR)
  • Local backup: Python Flask server on port 5000

How the Slot Refresh Loop Works
  1. 01
    Page Load → updateUI() fires immediately
    window.onload = updateUI ensures slot status is fetched as soon as the page loads — no stale data shown to the user.
  2. 02
    GET Request to Google Apps Script
    Fetch hits the GAS Web App URL. The Apps Script reads all rows from the Google Sheet and returns a JSON array. Each row has the slot number at index [3].
  3. 03
    Parse Occupied Slot Numbers
    data.map(r => r[3].toString()) extracts occupied slot numbers into a string array for O(1) lookup with Array.includes().
  4. 04
    Update DOM Slot Cards (1–3)
    Loop adds .occupied CSS class, sets status text to OCCUPIED, and removes the click handler for taken slots. Free slots get the handler restored and show FREE.
  5. 05
    setInterval — Auto-refresh every 30 s
    setInterval(updateUI, 30000) keeps the portal in sync with the physical parking lot without requiring a page reload.
Key Design Decisions
  • No-CORS POST mode: GAS endpoints don't return CORS headers for cross-origin requests from a local file. Booking POST uses mode: "no-cors" — the response body is opaque but the Sheets write still succeeds.
  • Fire-and-forget silent backup: saveTxtSilently() POSTs to a local Flask server with a 4 s AbortSignal timeout and .catch(() => {}) — any failure is silently swallowed so user experience is never affected.
  • QR encoded from plate, not booking ID: The QR code encodes the vehicle number plate directly, so both EasyOCR and the QR scanner reference the same string — no separate ID mapping layer needed.
  • CSS custom properties for theming: All brand colours defined as --primary, --success, --danger in :root — the entire palette can be changed from one declaration.
  • Regex plate validation: /^[A-Z]{2}[0-9]{2}[A-Z]{1,3}[0-9]{4}$/ enforces the standard Indian vehicle plate format before any API call is made, preventing junk data in the database.
